# Architecture Decision Records

## Overview

This document captures the key architectural decisions made for the SLSA Level 3 Supply Chain Security Lab, explaining the **why** behind each choice.

---

## ADR-1: Keyless Signing over Key-based Signing

### Context
Traditional image signing requires managing long-lived private keys (e.g., `cosign generate-key-pair`). This creates:
- Key rotation complexity
- Risk of key compromise
- Need for secure key storage (HSM, Vault, etc.)

### Decision
Use **Cosign keyless signing** with GitHub Actions OIDC tokens + Sigstore's public Fulcio CA.

### Rationale
1. **No key management**: Certificate is issued on-demand, valid for ~10 minutes
2. **Identity-based**: Tied to GitHub Actions workflow identity, not a static key
3. **Transparency**: All signatures recorded in Rekor transparency log (publicly auditable)
4. **Free**: Sigstore public instance is free and maintained by Linux Foundation

### Flow
```
GitHub Actions → OIDC JWT → Fulcio CA → Short-lived cert → Sign image → Record in Rekor
```

### Tradeoff
- Depends on Sigstore public infrastructure (99.9% uptime)
- For air-gapped environments, would need self-hosted Sigstore

## ADR-2: SLSA GitHub Generator as Isolated Builder

### Context
SLSA Level 3 requires provenance generated by a builder that the developer **cannot tamper with**. Running provenance generation as inline steps in your workflow does NOT meet Level 3 because the developer controls the workflow.

### Decision
Use `slsa-framework/slsa-github-generator` as a **reusable workflow** called in a separate job.

### Rationale
1. **Isolation**: The reusable workflow runs in GitHub's infrastructure, with its own permissions
2. **Non-falsifiable**: The developer cannot modify the provenance generation code
3. **Audited**: The generator is audited by the SLSA community
4. **Automatic**: Outputs SLSA v1.0 provenance format with correct builder ID

### Key Detail
```yaml
# This is a SEPARATE JOB, not inline steps
provenance:
  uses: slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@v2.1.0
```

## ADR-3: Distroless Base Image

### Context
Alpine is the most common "minimal" base image, but it still includes a shell (`/bin/sh`), package manager (`apk`), and ~500+ packages.

### Decision
Use `gcr.io/distroless/static-debian12:nonroot` as the runtime base image.

### Rationale
1. **No shell**: Attackers can't exec into the container
2. **No package manager**: Can't install additional tools
3. **Minimal SBOM**: Only ~20 components vs ~500+ for Alpine
4. **Reduced CVEs**: Dramatically fewer OS-level vulnerabilities
5. **Non-root by default**: UID 65532 built into the image

### Tradeoff
- Cannot debug by exec-ing into the container
- Solution: Use ephemeral debug containers (`kubectl debug`)

## ADR-4: Kyverno over OPA Gatekeeper

### Context
Both Kyverno and OPA Gatekeeper can enforce Kubernetes policies. Need to choose one for image verification.

### Decision
Use **Kyverno** for policy enforcement.

### Rationale
1. **Native YAML**: Policies are written in Kubernetes-native YAML (no Rego language to learn)
2. **Image verification built-in**: `verifyImages` rule type handles Cosign verification natively
3. **Mutation support**: Can auto-mutate image tags to digests (`mutateDigest: true`)
4. **Lower learning curve**: DevOps engineers already know YAML
5. **Active development**: Strong community, regular releases

### Example Comparison
```yaml
# Kyverno (simple YAML)
verifyImages:
- imageReferences: ["ghcr.io/rap1p1/*"]
  attestors:
  - entries:
    - keyless:
        issuer: "https://token.actions.githubusercontent.com"

# OPA Gatekeeper (requires Rego)
# Much more verbose, requires learning Rego language
```

## ADR-5: Digest-Pinned Image References

### Context
Tag-based image references (`:latest`, `:v1.0`) are mutable. An attacker could:
1. Compromise the registry
2. Replace the image behind the same tag
3. Your cluster pulls the compromised image

This is called a **tag mutation attack** or **TOCTOU (Time-of-check-time-of-use)** attack.

### Decision
All Kubernetes manifests reference images by **digest** (`@sha256:abc123...`), never by tag.

### Implementation
1. Pipeline outputs the exact digest after build
2. CD workflow updates deployment.yaml with the digest
3. Kyverno's `mutateDigest: true` also enforces this for any manual deployments

## ADR-6: CycloneDX over SPDX for Primary SBOM

### Context
Two major SBOM standards exist: CycloneDX and SPDX. Need to choose a primary format.

### Decision
Use **CycloneDX JSON** as the primary format. Also generate SPDX as secondary output.

### Rationale
1. **JSON-native**: CycloneDX was designed for JSON from the start
2. **Richer vulnerability data**: Better VEX (Vulnerability Exploitability eXchange) support
3. **Kyverno compatibility**: Better attestation verification support
4. **Industry momentum**: Increasingly adopted by OWASP, government mandates
5. **Tooling**: Syft, Trivy, and Grype all support CycloneDX natively

## ADR-7: Three-Policy Separation for Defense in Depth

### Context
Could implement all checks in a single Kyverno policy. However, this reduces visibility and makes troubleshooting harder.

### Decision
Separate into three distinct policies:
1. `require-signed-images` — Verify Cosign signature
2. `require-slsa-provenance` — Verify SLSA Level 3 provenance
3. `require-sbom` — Verify SBOM attestation

### Rationale
1. **Independent failure modes**: Know exactly which check failed
2. **Gradual rollout**: Can enable policies one at a time (start with Audit mode)
3. **Policy reports**: Separate reports for each policy type
4. **Compliance mapping**: Map each policy to specific compliance requirements
